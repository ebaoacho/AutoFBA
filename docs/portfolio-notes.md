# Portfolio Notes

このドキュメントは、LOGEX を就活や面接で説明するための要点整理です。

## 30 秒説明

Amazon FBA 出品者向けの在庫管理・発注支援アプリです。  
Next.js と Django で構築し、SKU ごとの在庫、販売速度、利益率、発注タイミングを一元管理できるようにしました。  
Amazon SP-API 連携、Cookie 認証、トークン暗号化保存まで含めて、実務寄りのフルスタック構成にしています。

## 1 分説明

FBA 運用では、SKU ごとに在庫量、販売速度、仕入価格、発注リードタイムをまとめて見ないと欠品や過剰在庫が起きやすいです。  
そこで、SKU 一覧、推奨発注数、利益率、外部 API 連携をまとめた Web アプリを作りました。  
フロントは Next.js、バックエンドは Django REST Framework、認証は Cookie ベース JWT、Amazon SP-API のトークンは暗号化して保存しています。  
単なる CRUD ではなく、業務判断ロジックをアプリに落とし込んだ点が特徴です。

## 面接で押さえるポイント

### 技術面

- フロントとバックエンドを責務分離している
- 外部 API 連携をバックエンド側に閉じ込めている
- Cookie 認証、CORS、CSRF を考慮している
- データ可視化と時系列データの扱いがある

### プロダクト面

- 誰のどんな課題を解くアプリかが明確
- 在庫判断という実務課題に対して機能が結びついている
- 一覧画面、分析画面、発注支援画面で利用シーンが想像しやすい

### 設計面

- `SKUConfig` を中心にデータをまとめている
- 履歴 API と分析 API を分けている
- フロント側で在庫データをマージして表示体験を作っている

## 改善余地として話せること

- テストコードが不足している
- ビュー層のロジックをサービス層に切り出す余地がある
- デプロイ用ディレクトリ構成を整理できる
- 文字コードや文言、環境変数管理をより厳密にしたい

## 深掘り質問への返し方

### 「一番工夫した点は？」

在庫一覧を表示するだけでなく、安全在庫日数とリードタイムから推奨発注数を出して、実際の判断に使える UI にした点です。

### 「難しかった点は？」

Amazon SP-API の認可や認証情報の扱いです。単に API を呼ぶだけでなく、トークンの保存方法や責務分離まで考える必要がありました。

### 「今後どう改善する？」

分析ロジックの整理、テスト追加、環境変数の安全な管理、画面キャプチャやデモ整備を進めると、より完成度の高いプロダクトになります。
